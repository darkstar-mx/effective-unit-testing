Smell 4 Assertion Roulette

Bad example

@Test
void createInvoice_ShouldPopulateFields_AndNotifyCustomer() {
    Invoice invoice = service.createInvoice(order);

    assertEquals("INV-001", invoice.id());
    assertEquals(CREATED, invoice.status());
    assertEquals(customerId, invoice.customerId());
    assertEquals(3, invoice.lines().size());
    assertEquals(total, invoice.total());

    verify(emailGateway).sendInvoice(order.customerEmail(), invoice);
}

Good example

@Test
void createInvoice_ShouldReturnCreatedInvoiceForOrder() {
    // Arrange
    Invoice expected = new Invoice("INV-001", CREATED, customerId, 3, total);

    // Act
    Invoice actual = service.createInvoice(order);

    // Assert
    assertAll("created invoice",
        () -> assertEquals(expected.id(), actual.id(), "invoice id"),
        () -> assertEquals(expected.status(), actual.status(), "invoice status"),
        () -> assertEquals(expected.customerId(), actual.customerId(), "customer"),
        () -> assertEquals(expected.lines().size(), actual.lines().size(), "line count"),
        () -> assertEquals(expected.total(), actual.total(), "invoice total")
    );
}

Cause 1: Eager Test

If the assertions answer different business questions, the test is doing too much and should be split into separate tests.

Cause 2: Missing Assertion Message

If several checks support one behavior, unnamed assertions force the reader to reverse-engineer what broke.

Fix 1

Split invoice creation and customer notification into separate tests because they are different behaviors.

Fix 2

When multiple checks support one behavior, use assertion messages or compare a richer expected object.

Review rule: split the test when assertions prove different behaviors; restructure the assertions when they prove one behavior poorly.
