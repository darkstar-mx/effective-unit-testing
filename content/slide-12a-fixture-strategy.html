Fixture Strategy Keep Test Data Explicit and Cheap

Bad example

@Test
void updateEmail_ShouldPersistChange() {
    User user = TestDataFactory.createDefaultEnterpriseUser();

    service.updateEmail(user, "new@email.com");

    assertEquals("new@email.com", user.getEmail());
}

Good example

@Test
void updateEmail_ShouldPersistChange() {
    // Arrange
    User user = aUser()
        .withId(1L)
        .withEmail("old@email.com")
        .build();

    // Act
    service.updateEmail(user, "new@email.com");

    // Assert
    assertEquals("new@email.com", user.getEmail());
}

Use inline setup when

  • The object is small.
  • The fields matter to the scenario.
  • Duplication is still readable.

Use a builder/factory when

  • The object has many fields.
  • You need safe defaults.
  • The test should highlight only the relevant differences.

Avoid

  • Mystery helpers with hidden defaults.
  • Shared mutable fixtures.
  • Factories that encode business meaning unclearly.

Review rule

The fixture should reveal why this scenario is special without making the reader inspect helper internals first.
