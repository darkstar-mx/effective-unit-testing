Code Review Checklist

Approve when

  • The test names a behavior.
  • The setup is minimal and relevant.
  • The assertion proves the rule.
  • The test survives harmless refactoring.
  • Mocks represent real boundaries.

Push back when

  • The test verifies internals.
  • The fixture is large or magical.
  • The assertion is vague.
  • Coverage is the only argument.
  • The test is harder to understand than the code.

A review comment on tests should be as rigorous as a review comment on production design.
