Why Unit Tests Exist And Why They Fail

Problem

  • Teams believe tests are about “coverage” or “correctness”.
  • Tests become a burden instead of a tool.
  • Engineers start avoiding or deleting tests.

Rule of Thumb

If your code is hard to test, your design is the problem.

class OrderService {
    double calculateTotal(Order order) {
        double base = order.getItems()
            .stream()
            .mapToDouble(Item::price)
            .sum();

        return base * 1.16;
    }
}
class OrderService {
    private final TaxPolicy taxPolicy;

    OrderService(TaxPolicy taxPolicy) {
        this.taxPolicy = taxPolicy;
    }

    double calculateTotal(Order order) {
        return taxPolicy.apply(order.subtotal());
    }
}

Good tests improve design, make refactoring safe, and reduce long-term maintenance cost.

1

2

3

Failures are brittle tests reacting to harmless internal changes.

The Real Problem Is Not Tests

Wrong Mental Model

  • Tests slow us down.
  • Testing is overhead.
  • CI is the bottleneck.

Better Mental Model

  • Good tests accelerate development.
  • Good tests protect refactoring.
  • Good tests document behavior.

Tests are valuable only if they make development faster.

Meaningless Assertions Signal vs Noise

@Test
void findOrder_ShouldReturnOrder() {
    Order result = service.findOrder(123L);

    assertNotNull(result);
}
@Test
void findOrder_ShouldReturnCorrectOrderDetails() {
    Order expected = new Order(123L, "PENDING", 500);

    Order actual = service.findOrder(123L);

    assertEquals(expected, actual);
}

Why it fails

No verification of actual behavior. Poor failure diagnostics.

Rule

Assertions should explain failures clearly.

What Makes a Test Valuable?

01Fast

Runs in milliseconds.

02Deterministic

Same result every time.

03Readable

Explains behavior.

04High Signal

Failures are useful.

@Test
void processOrder() {
    var result = service.process(order);

    assertTrue(result != null);
}
@Test
void processOrder_ShouldApplyDiscount_WhenUserIsPremium() {
    User user = new User(Status.PREMIUM);
    Order order = new Order(100);

    var result = service.process(user, order);

    assertEquals(90.0, result.total());
}

AAA Pattern Readability Through Structure

@Test
void testProcess() {
    Service s = new Service();
    s.init();
    assertTrue(s.isValid());

    var result = s.process(new Request(10));
    assertEquals(20, result.val());
}
@Test
void process_ShouldDoubleValue_WhenInputIsPositive() {
    // Arrange
    var service = new Service();
    var request = new Request(10);

    // Act
    var actual = service.process(request);

    // Assert
    assertEquals(new Response(20), actual);
}

Separate Arrange, Act, and Assert explicitly.

DAMP > DRY Clarity Over Duplication

@Test
void test1() {
    helper(true, false, "Standard");
}
@Test

    User user = new User("John", Status.ACTIVE);
    Order order = new Order(100);

Prefer descriptive duplication over hidden abstractions.

State Over Interaction What vs How

@Test
void saveUser_ShouldInteractWithDatabase() {
    userService.save(user);

    verify(mockDb).executeInsert(any());
}
@Test
void saveUser_ShouldMakeUserRetrievable() {
    userService.save(user);

    User actual = userService.findById(user.id());

    assertEquals(user, actual);
}

Verify outcomes, not internal interactions.

Wrap non-deterministic behavior behind injectable seams.

Mocks vs Stubs

Stub

Provides data to the system under test.

Mock

Verifies behavior or interaction.

// Stub
User stubUser = mock(User.class);
when(stubUser.id()).thenReturn(1L);

// Mock
EmailService mockEmail = mock(EmailService.class);
service.notify(stubUser);
verify(mockEmail).send(any());

Stubs provide data; mocks verify behavior.

Over-Mocking The False Confidence Trap

@Test
void processOrder_ShouldSucceed() {
    User mockUser = mock(User.class);

    when(mockUser.isActive()).thenReturn(true);

    orderService.process(mockUser, order);
}
@Test
void processOrder_ShouldSucceed_ForActiveUser() {
    User user = new User(
        1L, "John", Status.ACTIVE
    );

    orderService.process(user, order);
}

Prefer real objects unless crossing expensive boundaries.

Spies and the doReturn Gotcha

Service spy = spy(new Service());

when(spy.expensiveCall())
    .thenReturn(result);
Service spy = spy(new Service());

doReturn(mockResult)
    .when(spy)
    .expensiveCall();

Use doReturn with spies.

Spying on Overloads

@Test
void getTotalPrice_ShouldDelegateDefaultQuantity() {
    PriceService spy = spy(new PriceService());

    spy.getTotalPrice();

    verify(spy).getTotalPrice(1);
}

Test logic once; verify delegation separately.

ArgumentCaptor

@Captor
ArgumentCaptor<Request> requestCaptor;

@Test
void submit_ShouldSendCorrectPayload() {
    service.submit(data);

    verify(apiClient)
        .sendRequest(requestCaptor.capture());

    Request captured = requestCaptor.getValue();

    assertEquals("TX-123", captured.id());
}

Use ArgumentCaptor for deep payload validation.

Small Tests The Google Standard

@Test
void processFile_ShouldUpdateDatabase() {
    processor.process(new File("config.json"));
}
@Test
void processContent_ShouldGenerateValidResult() {
    String input = "{ \"key\": \"value\" }";

    var result = processor.process(input);

    assertEquals(expected, result);
}

Unit tests should avoid I/O entirely.

The Private Method Fallacy

    service.internalTransform("data");
}
@Test

    service.process("data");

    var saved = repo.findByName("DATA");

The InjectMocks Pitfall

public class BaseTest {
    public BaseTest() {
        MockitoAnnotations.openMocks(this);
    }
}
@ExtendWith(MockitoExtension.class)
class UserTest {

    @Mock
    UserRepo repo;

    @InjectMocks
    UserService service;
}

Use MockitoExtension, not constructors.

What “Good” Looks Like

01Fast CI

02Stable Tests

03Clear Failures

04Safe Refactoring

Change becomes routine.

@Test
void transfer_ShouldReject_WhenBalanceIsInsufficient() {

    var result = service.transfer(account, 200);

If You Remember Only This

  1. Test behavior, not implementation.
  2. Assert state, not interactions.
  3. Avoid logic in tests.
  4. Prefer DAMP over DRY.
  5. Use meaningful assertions.
  6. Keep tests deterministic.
  7. Test through public APIs.
  8. Prefer readability over cleverness.
  9. Tests should survive harmless refactors.
